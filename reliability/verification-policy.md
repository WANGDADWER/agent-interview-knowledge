# 资料可信度与核验规则

> 最后核验日期：2026-07-01  
> 目标：避免在面试知识库中写入虚假信息、过期信息或未经查证的生态判断。

## 一、信息分级

| 等级 | 类型 | 能否写入面试文档 | 处理方式 |
| --- | --- | --- | --- |
| A | 官方文档、官方 GitHub、Microsoft Learn、协议官网 | 可以作为事实写入 | 必须附来源链接 |
| B | 官方博客、官方发布说明、官方迁移指南 | 可以作为事实写入，但要注明时间 | 必须附来源链接和发布日期/核验日期 |
| C | 高质量技术博客、课程、视频、社区讨论 | 可作为学习参考 | 不写成事实，只写成“实践建议” |
| D | 未注明来源的二手文章、营销文章、旧教程 | 不作为依据 | 只用于发现线索，不直接引用 |

## 二、写入规则

1. **框架定位必须来自官方来源。** 例如 LangGraph、LlamaIndex、AutoGen、MCP 的定义必须从官方文档或官方 GitHub 核验。
2. **生态变化必须来自官方来源。** 例如 AutoGen 是否进入维护模式、是否建议新项目转向 Microsoft Agent Framework，只能以官方 GitHub/Microsoft Learn 为依据。
3. **源码路径不能凭记忆写死。** 由于开源仓库结构会变化，源码精读文档只写“核心符号/核心抽象/检索方法”，不随意写未经核验的具体文件路径。
4. **代码样例区分两类：**
   - 官方 API 示例：必须和官方文档一致，并标注来源；
   - 手写 mini 实现：明确说明是学习用简化版本，不代表官方源码实现。
5. **工程建议必须标注为建议。** 例如“生产中建议加审批、审计、幂等键”属于工程建议，不能伪装成某框架官方能力。

## 三、更新流程

每次更新文档时，按下面流程执行：

```text
提出结论
→ 找官方来源
→ 记录链接和核验日期
→ 判断是否为事实/建议/推论
→ 写入文档
→ 面试表达中避免夸大
```

## 四、禁止写法

- “某框架已经完全取代某框架”——除非官方明确这样表述。
- “某模块一定在某文件路径”——除非已核验当前仓库源码。
- “大厂一定会问某题”——只能写成“高频可能追问”。
- “某方法一定提升效果”——没有实验结果时只能写成“可尝试方向”。

## 五、推荐写法

- “根据官方文档，LangGraph 定位为低层编排框架和运行时，面向 long-running、stateful agents。”
- “根据 AutoGen 官方 GitHub，AutoGen 目前处于 maintenance mode；新项目官方建议从 Microsoft Agent Framework 开始。”
- “以下 mini 实现用于理解机制，不是官方源码复刻。”
